# Rvbbit Route Profiles

## Native Router Catalog

`pg_rvbbit` now has a native routing control plane. It does not execute DuckDB
itself yet; it stores adaptive profiles, extracts query-shape features inside
Postgres, and explains the route the production executor should take once the
Rust Duck path is wired in.

Fresh extension installs create:

- `rvbbit.route_profiles`: named JSON profiles, with at most one active profile.
- `rvbbit.route_observations`: per-query candidate timing observations.
- `rvbbit.route_profile_entries`: indexed exact-shape decisions extracted from
  active/imported profiles.
- `rvbbit.route_profile_points`: indexed row-count curve anchors extracted from
  offline profiles.
- `rvbbit.route_observation_summary`: candidate timing medians/p95 by shape.
- `rvbbit.route_shape_summary`: winner, gain, and exploration gaps by shape.
- `rvbbit.route_profile_summary`: active/inactive profile decisions in a UI
  friendly shape.
- `rvbbit.route_features(sql text)`: extract the native feature vector.
- `rvbbit.route_explain(sql text)`: explain candidate choice, fallback, and why.
- `rvbbit.route_explain_text(sql text)`: readable route explanation for psql/UI
  debugging.
- `rvbbit.route_profiles()`: list installed profiles with entry/point counts
  and candidate mix.
- `rvbbit.route_status()`: one-call install health payload: current profile,
  candidate gates, runtime fail-open posture, and catalog row counts.
- `rvbbit.route_record_observation(...)`: log one measured candidate timing.
- `rvbbit.route_train(name, min_observations, min_gain_pct)`: fit and activate a
  profile from catalog observations.
- `rvbbit.route_eval(name)`: summarize profile coverage and online exploration
  gaps.
- `rvbbit.route_set_profile(name, profile, active)`: load an offline profile.
- `rvbbit.route_export_profile(name)`: emit a complete portable profile with
  exact entries and row-count curve points.
- `rvbbit.route_import_profile(name, profile, active)`: restore a portable
  profile into the indexed catalog tables.

Load the current offline profile into the native catalog:

```bash
docker compose -f docker/docker-compose.yml -f docker/docker-compose.competitors.yml \
  exec -T bench python /bench/rvbbit_route_load_profile.py \
  --profile /bench/rvbbit_route_profile.json \
  --name bench-combined
```

Inspect a route decision without running the query:

```sql
SELECT jsonb_pretty(
  rvbbit.route_explain('select count(distinct "SearchPhrase") from hits')
);
```

Fresh installs are deliberately conservative. If no route profile is active,
the normal backend keeps Rvbbit-table queries on the native PostgreSQL/Rvbbit
path after hard-rule rewrites. Import and activate a profile before expecting
Duck/DataFusion routing:

```sql
SELECT jsonb_pretty(rvbbit.route_status());
SELECT rvbbit.route_activate_profile('bench-combined');
SELECT jsonb_pretty(rvbbit.route_current_profile());
```

The Duck/DataFusion backend is fail-open by default. If a selected sidecar
engine errors at runtime, `rvbbit.duck_backend_fail_open=on` reruns the original
query through the native PostgreSQL/Rvbbit path with `rvbbit.duck_backend=off`
for that nested execution. Set `RVBBIT_DUCK_BACKEND_FAIL_OPEN=0` or
`SET rvbbit.duck_backend_fail_open = off` when debugging executor failures and
you want them to surface as query errors.

The native shape keys intentionally mirror the Python prototype: SHA-256 query
fingerprints, 8-character clause signatures, row-count buckets, and native
function detection from PostgreSQL `EXPLAIN`. That lets profiles generated by
the benchmark harness seed the native router instead of starting cold.

Profiles are no longer routed by walking a large JSON document. Imported JSON
is compacted into metadata, while exact choices live in
`route_profile_entries` and interpolation anchors live in `route_profile_points`.
That keeps route lookup index-friendly and gives the UI stable tables to query.

The intended adaptive loop is:

1. Route with hard rules plus the active profile.
2. Record candidate timings for safe SELECTs.
3. Periodically call `rvbbit.route_train(...)` from a maintenance job.
4. Keep the new profile active only when it clears minimum observation and
   gain thresholds.

`pg_rowstore` is represented as a third candidate in explanations, but remains
unavailable until we deliberately add a heap/index sidecar. For now the live
choices are `duck_vector` and `rvbbit_native`.

## Offline Training

The Duck hot path can now consult an explainable routing profile instead of
only using fixed rules. Profiles are generated from benchmark runs that include
both forced native and forced Duck paths:

```bash
BENCH_LIMIT=500000 BENCH_QUERIES=Q20,Q21,Q22,Q23,Q24,Q25,Q26 \
  ./bench/clickbench/build_rvbbit_route_profile.sh
```

Use a profile in auto mode:

```bash
RVBBIT_ROUTE_PROFILE=/bench/rvbbit_route_profile.json \
RVBBIT_ROUTE_LOG=/bench/clickbench/results/route_trace.jsonl \
BENCH_SYSTEMS=rvbbit ./bench/clickbench/run_offline.sh
```

The guarded Duck runtime treats `rvbbit.route_explain(...)` as the source of
truth in auto mode. Python is now only the executor bridge: it asks Rust for
the route, runs either the native PostgreSQL/Rvbbit path or DuckDB, then records
the measured result. Control that bridge with:

- `RVBBIT_NATIVE_ROUTER=auto` default: use native router if available.
- `RVBBIT_NATIVE_ROUTER=off`: skip the native router bridge and use the guarded
  Duck fallback behavior.
- `RVBBIT_NATIVE_ROUTER=require`: fail the Duck bridge if native routing is not
  available.
- `RVBBIT_ROUTE_OBSERVE=1` default: record chosen-path timings into
  `rvbbit.route_observations`, so later `rvbbit.route_train(...)` has online
  workload data. Use `RVBBIT_ROUTE_OBSERVE=0` to disable catalog writes.
  Forced native/forced Duck benchmark runs are still the better source for
  unbiased two-candidate training data.
- `RVBBIT_ROUTE_EXPLORE_PCT=0.01`: opt into controlled exploration. For safe
  SELECTs with both native and Duck available, the bridge occasionally executes
  the non-chosen candidate and records it with `source=explore:...`. The chosen
  exploratory path is the returned result; this is not shadow execution.
- `RVBBIT_ROUTE_DUCK_VECTOR`, `RVBBIT_ROUTE_DATAFUSION_MEM`,
  `RVBBIT_ROUTE_DATAFUSION_VECTOR`,
  `RVBBIT_ROUTE_DUCK_HIVE`, `RVBBIT_ROUTE_DATAFUSION_HIVE`,
  `RVBBIT_ROUTE_HIVE`, `RVBBIT_ROUTE_PG_ROWSTORE`, and
  `RVBBIT_ROUTE_RVBBIT_NATIVE`: set any to `0`/`off` to remove that candidate
  from training/runtime routing. `RVBBIT_ROUTE_HIVE=0` disables both Hive
  candidates without disabling normal parquet scans.
- `RVBBIT_ROUTE_HIVE_MIN_CONFIDENCE=0.08` default: require a stronger predicted
  win before routing to a Hive layout, since it costs extra write/storage work.
- `RVBBIT_ROUTE_NO_PROFILE_NATIVE_MAX_ROWS=500000` default: keep small/simple
  no-profile analytical shapes on the native path unless the shape looks like a
  better vector/variant candidate.
- `RVBBIT_ROUTE_NO_PROFILE_VARIANT_MIN_ROWS=250000` default: allow no-profile
  fallback and profile-miss fallback to choose Hive parquet variants for
  text-heavy distinct/top-k/rollup shapes once variants are available.
- `RVBBIT_DUCK_BACKEND_FAIL_OPEN=1` default: runtime Duck/DataFusion failures
  fall back to native PostgreSQL/Rvbbit execution instead of failing the user
  query.
- `RVBBIT_PARQUET_META_CACHE=1` default: the Rust Duck/DataFusion sidecar
  keeps an in-memory catalog snapshot keyed by an authoritative Postgres
  row-group/layout fingerprint. Set to `0`/`off` to force a fresh catalog load
  for every sidecar request.
- `RVBBIT_PARQUET_PREWARM=1` default: when the sidecar builds or refreshes an
  executor, it reads parquet footer/schema metadata into an in-memory cache
  before creating DuckDB/DataFusion views. This does not scan column data.
- `RVBBIT_ROUTE_SAFETY_CACHE=1` default: persistent Rust sidecars cache exact
  SQL `rvbbit.route_explain(...)` safety approvals under the current parquet
  catalog fingerprint. The cache is invalidated on row-group/layout/schema,
  heap-tail, or delete-log changes. Set `RVBBIT_ROUTE_SAFETY_CACHE=0` to force
  a fresh safety check on every sidecar request. `RVBBIT_ROUTE_SAFETY_CACHE_MAX`
  caps entries per sidecar process (default `4096`).
- `RVBBIT_ROUTE_SAFETY_LOCAL=1` default: before calling
  `rvbbit.route_explain(...)`, the sidecar locally approves conservative
  `FROM`/`JOIN` SELECTs whose referenced relations all resolve to authoritative
  Rvbbit parquet catalog entries. Complex SQL falls back to `route_explain`.
- `RVBBIT_DUCK_RUST_PERSISTENT=1` default for offline forced benchmark paths:
  `rvbbit_duck_forced`, `rvbbit_datafusion_mem_forced`,
  `rvbbit_datafusion_forced`, and Hive variants keep one
  Rust sidecar process per engine/layout for the duration of the Python
  benchmark runner. Set to `0` to restore one process per query.

The profile is keyed by query shape features, not benchmark query ids. At
runtime, matching checks native hard rules, exact profile entries, imported
profile curve points, and online observation curves before falling back to the
default Duck vector candidate. Set `RVBBIT_ROUTE_TRACE=1` to include SQL text in
the JSONL trace.

Useful catalog probes for a UI:

```sql
SELECT * FROM rvbbit.route_shape_summary
ORDER BY needs_exploration DESC, last_seen DESC NULLS LAST;

SELECT jsonb_pretty(rvbbit.route_eval('bench-combined'));
```

To rebuild the full offline training corpus from forced executor timings, use
the orchestration script:

```bash
./bench/rebuild_rvbbit_route_training.sh
```

By default it runs ClickBench at `5000,50000,500000,5000000` rows and TPC-H at
scale factors `0.05,0.1,0.33,1`, using `rvbbit_native_forced`,
`rvbbit_duck_forced`, `rvbbit_duck_hive_forced`,
`rvbbit_datafusion_mem_forced`, `rvbbit_datafusion_forced`,
`rvbbit_datafusion_hive_forced`, and
`rvbbit_pg_heap_forced`, then merges everything into
`bench/rvbbit_route_profile.json`. `rvbbit_native_forced` forces
`rvbbit.route_force_candidate=rvbbit_native`; legacy `rvbbit_native` is still
available as the `rvbbit.duck_backend=off` baseline. Set
`RVBBIT_ROUTE_INCLUDE_HIVE=0` to skip Hive forced paths and avoid building Hive
layout variants during training. It
also archives each raw forced-timing
result as `rvbbit_route_results.*.json` next to the suite profile. Useful knobs:

```bash
DRY_RUN=1 ./bench/rebuild_rvbbit_route_training.sh

CLICKBENCH_LIMITS="50000 500000" TPCH_SCALES="0.1" BENCH_REPEATS=1 \
  ./bench/rebuild_rvbbit_route_training.sh

RVBBIT_ROUTE_IMPORT=1 ./bench/rebuild_rvbbit_route_training.sh
```

TATP is intentionally skipped by this script for now: it is transactional and
does not produce comparable forced route timings for SELECT routing.

You can also merge already-produced scale runs into one profile:

```bash
docker compose -f docker/docker-compose.yml -f docker/docker-compose.competitors.yml \
  exec -T bench python /bench/rvbbit_route_train.py \
  --suite clickbench \
  --results /bench/clickbench/results/rvbbit_route_results.clickbench_5000.json \
  --results /bench/clickbench/results/rvbbit_route_results.clickbench_50000.json \
  --results /bench/clickbench/results/rvbbit_route_results.clickbench_500000.json \
  --results /bench/clickbench/results/rvbbit_route_results.clickbench_5000000.json \
  --output /bench/clickbench/results/rvbbit_route_profile.clickbench_multi.json
```

TPC-H uses the same route logger through `rvbbit_duck_hot` and is the next
best source of distinct query shapes:

```bash
TPCH_SCALE=0.1 BENCH_REPEATS=3 \
  ./bench/tpch/build_rvbbit_route_profile.sh
```

Route-log a TPC-H auto run:

```bash
TPCH_SCALE=0.1 SKIP_LOAD=1 BENCH_SYSTEMS=rvbbit \
  RVBBIT_ROUTE_PROFILE=/bench/rvbbit_route_profile.json \
  RVBBIT_ROUTE_LOG=/bench/tpch/results/route_trace.jsonl \
  ./bench/tpch/run_offline.sh
```

Keep ClickBench and TPC-H as separate training outputs, then merge them:

```bash
./bench/build_combined_rvbbit_route_profile.sh \
  bench/clickbench/results/rvbbit_route_profile.clickbench_multi.json \
  bench/tpch/results/rvbbit_route_profile.tpch_sf0_1.json
```

Do not pass the combined output profile back in as an input profile. The merge
script refuses that now because it can duplicate stale observations.

Evaluate a profile's observed coverage/regret:

```bash
python bench/rvbbit_route_eval.py --profile bench/rvbbit_route_profile.json
```

The catalog profile lifecycle is available through native SQL helpers. These
copy both exact shape entries and row-count interpolation points, so they are
safer than hand-written `INSERT ... SELECT` maintenance:

```sql
SELECT rvbbit.route_clone_profile('bench-combined', 'bench-combined-copy', false);

SELECT rvbbit.route_merge_profiles(
  'bench-combined-plus-online',
  '["bench-combined", "online-clickbench"]'::jsonb,
  true
);

SELECT rvbbit.route_activate_profile('bench-combined');
SELECT rvbbit.route_retire_profile('bench-combined-copy');
```

Back up and restore profiles from the bench container:

```bash
docker compose -f docker/docker-compose.yml -f docker/docker-compose.competitors.yml \
  exec -T bench python /bench/rvbbit_route_profile_admin.py \
  export --active --output /bench/results/active_route_profile.json

docker compose -f docker/docker-compose.yml -f docker/docker-compose.competitors.yml \
  exec -T bench python /bench/rvbbit_route_profile_admin.py \
  import --profile /bench/results/active_route_profile.json \
  --name restored-bench-combined --active
```

`route_set_profile` still exists for compatibility, but new automation should
prefer `route_import_profile` because it is explicit about portable profile
restore semantics.

Evaluate an auto-routing run against forced native and forced Duck:

```bash
python bench/rvbbit_route_eval.py --results bench/clickbench/results/last_run.json
```

Explain a route decision for an arbitrary query without executing the query:

```bash
docker compose -f docker/docker-compose.yml -f docker/docker-compose.competitors.yml \
  exec -T bench python /bench/rvbbit_explain_route.py \
  --query 'select count(distinct "SearchPhrase") from hits'
```

This is the current prototype for a future `EXPLAIN (RVBBIT)` surface. It
runs PostgreSQL `EXPLAIN`, checks whether referenced tables use the Rvbbit
access method, applies the route profile, optionally asks DuckDB to `EXPLAIN`
the translated parquet views, and reports `route=none` for non-Rvbbit or
non-read-only queries. The output is already shaped as a three-candidate
router:

- `duck_vector`: DuckDB over Rvbbit parquet row groups.
- `duck_hive`: DuckDB over Hive-partitioned Rvbbit parquet variants.
- `datafusion_vector`: DataFusion over Rvbbit parquet row groups.
- `datafusion_hive`: DataFusion over Hive-partitioned Rvbbit parquet variants.
- `rvbbit_native`: PostgreSQL executor over Rvbbit native rewrites/custom scans.
- `pg_rowstore`: retained shadow heap rowstore path, mainly useful as insurance
  and for small query shapes where PostgreSQL wins.

TATP is not a good Duck-routing corpus. It is mostly transactional DML and
point reads, so it is still useful for correctness/contract testing, but not
for choosing between native vector execution and Duck parquet execution.
